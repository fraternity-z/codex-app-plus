# 删除深度耦合功能模块的完整流程

## 困境

在删除 Codex App Plus 内置提供商配置功能时遇到以下挑战：

- **功能冲突**：内置提供商配置功能与 CC Switch 冲突，需要完全删除
- **规模庞大**：功能涉及前后端 30+ 文件，~2700 行代码
- **深度耦合**：与认证模式管理深度耦合，删除时需要保留部分功能（只读显示、ChatGPT 切换、OAuth 快照）
- **缺乏流程**：缺乏系统化的删除流程，容易遗漏或误删关键代码

## 策略

### 1. 探索阶段：全面分析功能范围

**目标**：彻底了解功能的实现范围和依赖关系

**具体步骤**：
- 使用 `Grep` 工具搜索关键词（如 `codex_provider`, `CodexProvider`）定位所有相关文件
- 使用 `Glob` 工具按模式匹配文件（如 `**/*provider*`）
- 分析模块依赖关系：前端 Bridge → 后端 Tauri 命令 → Rust 模块
- 识别深度耦合点（如 `codex_auth` 与 `codex_provider` 的关系）
- 参考竞品（CC Switch）理解冲突原因和替代方案

**产出**：`exploration-report.md`，包含：
- 功能规模统计（代码行数、文件数量）
- 完整的文件清单（后端、前端、测试、样式、国际化）
- 模块依赖关系图
- 冲突分析和替代方案建议

### 2. 设计阶段：明确删除与保留边界

**目标**：设计清晰的接口边界，避免误删或遗漏

**具体步骤**：
- 创建详细的文件清单，分类为：删除 / 保留 / 修改
- 设计接口边界：
  - 保留的接口：`getCodexAuthModeState()`, `activateCodexChatgpt()`, `captureCodexOauthSnapshot()`
  - 删除的接口：`listCodexProviders()`, `upsertCodexProvider()`, `deleteCodexProvider()`, `applyCodexProvider()`
- 简化数据模型：删除 `activeProviderId` 字段，保留 `activeProviderKey` 字段
- 规划替代方案：创建 CC Switch 推荐卡片，引导用户使用外部工具

**产出**：`plan.md`，包含：
- 17 个详细的实现步骤
- 接口边界设计
- 数据模型变更
- 验收标准

### 3. 协作阶段：spec-writer 与 spec-tester 协作

**目标**：通过协作确保设计合理性和测试覆盖

**具体步骤**：
- spec-writer 完成 `plan.md` 草稿后，提出接口边界问题：
  - 保留的 Bridge 接口是否足够？
  - 简化的数据模型是否影响现有功能？
  - 外部链接打开方式如何实现？
- spec-tester 审查设计并回答问题：
  - 确认接口设计合理性
  - 补充验收标准（边界情况、回归测试）
  - 设计测试用例（22 个用例，覆盖 5 个维度）
- 共同产出协作文档

**产出**：
- `test-plan.md`：22 个测试用例
- `interface-boundary-review.md`：接口边界审查报告
- `collaboration-summary.md`：协作总结

### 4. 实现阶段：按步骤严格执行

**目标**：按照 plan.md 的 17 个步骤严格执行，降低风险

**执行顺序**（关键）：
1. **先删除后端模块**（避免前端引用错误）
   - 删除 `src-tauri/src/codex_provider/` 整个目录
   - 删除 Tauri 命令和类型定义
   - 简化 `codex_auth` 模块
2. **再删除前端组件和配置逻辑**
   - 删除 UI 组件（`CodexProviderDialog`, `CodexProviderList`）
   - 删除配置逻辑（`codexProviderConfig.ts`）
   - 删除 Bridge 接口
3. **清理国际化和样式文件**
   - 删除 37 个国际化键，新增 15 个键
   - 删除 74 行 CSS 样式
4. **更新测试文件**
   - 删除提供商相关测试用例
   - 修复受影响的测试文件

**产出**：`summary.md`，包含：
- 完成的工作清单
- 文件统计（删除 7 个，新增 1 个，修改 15 个）
- 编译验证结果

### 5. 测试阶段：22 个测试用例全覆盖

**目标**：确保删除功能完整，保留功能正常，无回归问题

**测试维度**：
- **删除验证**（5 个用例）：确认代码完全移除，无残留
- **保留功能**（5 个用例）：确认只读显示、ChatGPT 切换、OAuth 快照正常
- **新增功能**（4 个用例）：验证 CC Switch 推荐卡片和外部链接
- **回归测试**（5 个用例）：确保 ChatGPT 登录、设置页面、应用启动正常
- **边界测试**（3 个用例）：处理空配置、无效 providerKey、混合配置

**验证方法**：
- 代码审查：检查文件是否删除、接口是否移除
- 编译检查：TypeScript 类型检查、Rust 编译
- 测试套件：运行 `pnpm test` 和 `cargo test`

**产出**：`test-report.md`，包含：
- 22 个测试用例的执行结果（100% 通过率）
- 发现的问题（本次无问题）
- 测试覆盖率分析
- 发布建议

## 理由

### 为什么这个策略有效

1. **系统化流程避免遗漏和误删**
   - 探索阶段的全面分析确保不遗漏任何相关文件
   - 详细的文件清单和分类避免误删保留功能

2. **探索阶段的全面分析为后续决策提供依据**
   - 理解模块依赖关系后，可以设计合理的接口边界
   - 参考竞品（CC Switch）提供替代方案思路

3. **接口边界协作确保设计合理性**
   - spec-writer 和 spec-tester 的协作发现潜在问题
   - 提前补充验收标准和测试用例，避免实现后返工

4. **分步执行降低风险，便于回滚**
   - 先删除后端再删除前端，避免引用错误
   - 每个步骤都有明确的验收标准
   - 出现问题时可以快速定位和回滚

5. **全面测试确保质量**
   - 22 个测试用例覆盖删除、保留、新增、回归、边界 5 个维度
   - 100% 通过率确保功能符合预期
   - 编译检查确保代码质量

### 关键成功因素

- **充分的探索分析**：投入时间全面了解功能范围
- **清晰的接口边界**：明确删除与保留的边界
- **多角色协作**：spec-writer 和 spec-tester 的协作提高设计质量
- **严格的执行顺序**：先后端后前端，降低风险
- **全面的测试覆盖**：确保无遗漏和回归问题
