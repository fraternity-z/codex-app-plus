# 账户额度显示功能 - 测试计划

## 1. 测试目标

验证账户额度显示功能在各种场景下的正确性、稳定性和用户体验，确保：
- 数据展示准确无误
- 边界情况优雅降级
- 交互功能正常工作
- 国际化支持完整
- 性能符合要求

## 2. 验收标准

### 2.1 功能验收标准

| 验收项 | 判定标准 | 优先级 |
|--------|---------|--------|
| 额度数据展示 | 5 小时限额和周限额百分比显示正确，误差 ≤ 1% | P0 |
| 重置时间显示 | 重置时间格式正确，倒计时准确 | P0 |
| 显示模式切换 | "已使用"/"剩余"模式切换正常，百分比计算正确 | P0 |
| 手动刷新 | 刷新按钮触发数据更新，加载状态正确显示 | P0 |
| 边界情况处理 | 所有边界情况（null 数据、部分字段缺失）不崩溃，优雅降级 | P0 |
| 国际化支持 | 中英文切换正常，所有文本正确翻译 | P0 |
| 性能要求 | 刷新操作响应时间 < 2 秒，组件渲染流畅 | P0 |

### 2.2 通过/不通过判定

**通过条件**：
- 所有 P0 优先级测试用例通过
- 无阻塞性 bug
- 代码覆盖率 > 80%
- 性能指标达标

**不通过条件**：
- 任何 P0 用例失败
- 存在数据展示错误或崩溃
- 性能指标不达标

## 9. 与 spec-writer 的讨论要点

需要与 spec-writer 确认的接口边界和异常情况：

### 9.1 ViewModel 层接口

**问题 1**: `buildAccountLimitCards()` 函数的返回值
- 当 `rateLimits === null` 时，返回空数组 `[]` 还是 `null`？
- 建议：返回空数组 `[]`，便于组件层统一处理

**问题 2**: 时间格式化函数的边界处理
- `formatResetTime(null)` 应该返回什么？"Unknown" 还是空字符串？
- 建议：返回 "Unknown"，提供明确的占位文本

**问题 3**: 百分比计算的精度
- 是否需要四舍五入？保留几位小数？
- 建议：保留整数，四舍五入（如 45.6% → 46%）

### 9.2 组件层接口

**问题 4**: `AccountLimitCard` 的 badge 显示逻辑
- badge 和 value 是否可以同时显示？
- 建议：badge 优先，如果有 badge 则不显示 value

**问题 5**: 刷新失败时的错误提示
- 错误提示显示在哪里？Toast 还是卡片内？
- 建议：使用 Toast 提示，不影响卡片布局

### 9.3 边界情况处理

**问题 6**: `primary` 和 `secondary` 都为 null 时
- 是否隐藏整个 AccountLimitsSection？
- 建议：隐藏，与 `rateLimits === null` 行为一致

**问题 7**: `usedPercent` 超出 0-100 范围时
- 是否需要做边界限制？
- 建议：限制在 0-100 范围内，避免显示异常
